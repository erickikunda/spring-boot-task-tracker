01 — Domain Choice & Justification

The domain is a Task Tracker — users belong to projects, projects contain tasks, tasks have comments. Not a Todo app (too trivial — one entity, no relationships) and not e-commerce (too many concepts: inventory, payments, SKUs). A task tracker hits the sweet spot:

RequirementHow This Domain Satisfies It
Multiple JPA relationship typesManyToOne, OneToMany, ManyToMany (project membership) all appear naturally
Role-based authorizationADMIN / MANAGER / MEMBER roles map to real access rules without being contrived
Meaningful validationDue dates, status transitions (can't reopen a CANCELLED task), priority enum, required fields
Pagination & filteringTasks per project, filter by assignee / priority / status — realistic query patterns
Audit trailcreatedAt / updatedAt on every entity; who-reported-vs-who-assigned distinction
React-readyBoard-style UIs (Kanban) are the canonical React showcase — the frontend story writes itself

The name we'll use throughout: TaskFlow. The base package is com.example.taskflow, but we'll rename demo3 in Module 1.

Why a Strict Three-Layer Architecture?

The controller → service → repository split isn't just textbook ideology — it encodes real constraints. Controllers know about HTTP and JSON, services know about transactions and business rules, repositories know about SQL. When you violate this (a controller calling a repository directly, a service returning HttpStatus) you get coupling that makes testing painful and refactoring dangerous. The rule of thumb: if you'd have to spin up a servlet container to test it, it belongs in a controller; if you'd have to mock a Connection, it belongs in a repository.

Package-by-Layer vs Package-by-Feature

This project uses package-by-layer (domain/, service/, web/) rather than package-by-feature (project/ containing its own entity + service + controller). For a learning project this is intentional — it makes the architectural boundaries explicit and obvious. In a real microservices codebase you'd almost certainly go package-by-feature or you'd already be splitting into separate services. Both approaches are valid; the choice is about where you put friction.

04 — Technology Justifications

Maven over Gradle

Gradle is more powerful and faster for large multi-module builds. Maven wins here because the POM is declarative XML — every Spring Boot tutorial, Stack Overflow answer, and enterprise codebase from the last 15 years uses Maven. Reading a Gradle build requires knowing Groovy/Kotlin DSL; a POM is just data. When you're learning Spring, you don't want the build tool competing for your attention.

Spring Boot 4.0

Spring Boot 4.0 is built on Spring Framework 7 and requires Java 21. Notable implications for this project:

  • All imports are jakarta.* (not javax.*) — Jakarta EE 11
  • spring-boot-starter-webmvc (not spring-boot-starter-web) — renamed in Boot 4.x
  • Virtual threads (Project Loom) are first-class via spring.threads.virtual.enabled=true
  • Bean Background Initialization improves startup time significantly
Action Required

The generated POM has <java.version>17</java.version>. Spring Boot 4.0 requires Java 21. We'll fix this in Module 1 along with renaming the artifact and adding the remaining dependencies (H2, PostgreSQL, Security, JWT, Flyway, Validation).

H2 (dev) + PostgreSQL (prod)

H2 lets you run the entire app with ./mvnw spring-boot:run — no Docker, no setup, no state. Every developer gets a clean slate on each restart. PostgreSQL in prod is non-negotiable because H2's SQL dialect diverges enough from Postgres that it masks bugs (window functions, JSON operators, generated columns). The profile split enforces this gap is explicit and not accidentally bridged.

Flyway over Liquibase

Liquibase is more powerful (supports rollbacks, generates diff reports). Flyway wins because migrations are plain SQL — you write CREATE TABLE, not <createTable> XML. When something goes wrong in prod you're already looking at familiar SQL, not a tool-specific DSL. Flyway's conventions (sequential version numbers, checksums on applied migrations, flyway_schema_history table) are also easy to reason about.

JWT (JJWT library)

JJWT (io.jsonwebtoken) is the de facto standard for JWT in the Java ecosystem — well-maintained, Spring-agnostic, and the API is straightforward. We use stateless JWTs (no session state, no Redis, no sticky sessions) because:

  • Horizontally scalable without shared session storage
  • Self-contained — the server can validate a token without a DB lookup on every request
  • Teaches the core concepts of claims, signatures, and expiry without magic

The tradeoff: JWTs can't be revoked before expiry without extra infrastructure (a token denylist, usually Redis). For this learning lab that's acceptable; in prod you'd want short expiry + refresh tokens.

Lombok

Eliminates boilerplate (@Getter, @Builder, @RequiredArgsConstructor) at compile time via annotation processing. The risk: IDE support occasionally breaks after upgrades, and @Data on JPA entities is a trap (generates equals/hashCode based on all fields, which breaks when entities are proxied or partially loaded). Rule: never use @Data on a JPA entity — use explicit @Getter @Setter and a hand-written (or @NaturalId-based) equals/hashCode.

Relationship Decisions

Why UUID primary keys, not Long?

UUIDs prevent ID enumeration attacks (GET /tasks/1, /tasks/2 …), work in distributed systems where multiple nodes generate IDs, and are compatible with client-side ID generation (useful for offline-first apps). The tradeoff: 16 bytes vs 8 bytes, and index fragmentation is slightly worse. For this scale: irrelevant. Use UUID.randomUUID() and annotate with @GeneratedValue(strategy = GenerationType.UUID) (native JPA 3.1+ support — no more custom generators).

Why separate reporter and assignee?

They're different roles. The person who created the ticket isn't always the one working on it. Having both lets you implement "my reported tasks" and "my assigned tasks" views independently, and it's a real query pattern you'll need to practice: find all tasks where reporter = me OR assignee = me.

Why a join table for Project members?

Users belong to many projects; projects have many members. The join table project_members (project_id, user_id) is the correct JPA @ManyToMany pattern. If you later need metadata on the relationship (e.g., when the user joined, their role within the project), you promote the join table to a first-class entity (ProjectMembership) — a common real-world evolution. We'll model it simply for now.

Soft delete strategy

We won't implement soft deletes in this project. Soft delete (adding deleted_at and filtering every query) is a trap that leaks complexity everywhere and is usually the wrong solution — proper audit logging, event sourcing, or a separate "archived" state are better approaches for most domains. The ARCHIVED project status already handles the main use case.

06 — Spring Profile Strategy

Spring profiles are not environments — they're configuration overlays. The base application.properties holds settings that never vary; profiles hold things that do. The merge order is: base → active profile(s) → environment variables. Environment variables always win. This is the 12-Factor App model.

The open-in-view=false Gotcha

Spring Boot defaults spring.jpa.open-in-view to true, which keeps a database connection open for the entire HTTP request lifecycle, allowing lazy loading in the view layer. This is an anti-pattern: it silently causes N+1 queries in serialization, monopolizes connection pool threads during slow client reads, and hides design problems. Set it to false on day one and let the LazyInitializationExceptions teach you where to add explicit fetches or @Transactional.

07 — Docker Compose Strategy

Docker Compose serves one purpose here: spin up PostgreSQL locally so you can run the app against a real database without installing Postgres natively. The app itself does not run in Docker during development — that adds rebuild friction that hurts the feedback loop. Docker Compose is purely infrastructure.

docker-compose.yml (skeleton)

services:
  postgres:
    image: postgres:16-alpine
    container_name: taskflow-db
    environment:
      POSTGRES_DB: taskflow
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: changeme
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d taskflow"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

volumes:
  pgdata:

Usage

# Start PostgreSQL in the background
docker compose up -d

# Watch startup and confirm healthy
docker compose ps

# Connect directly (useful for debugging migrations)
docker exec -it taskflow-db psql -U appuser -d taskflow

# Stop (data persists in pgdata volume)
docker compose stop

# Nuke everything including the data volume
docker compose down -v
Why postgres:16-alpine?

Alpine-based images are ~80MB vs ~400MB for the full Debian variant. The tradeoff is that Alpine uses musl instead of glibc, which can cause subtle issues with some native extensions — but for a pure JDBC connection that doesn't matter. Always pin a major version (16 not latest) so the behavior doesn't silently change when a new major is released.

The pgdata Volume Decision

Using a named volume (not a bind mount to ./data) keeps Postgres data outside the project directory, preventing it from being accidentally committed, deleted by git clean -fd, or scanned by IDE indexers. The named volume is managed by Docker and survives docker compose down but is cleared by docker compose down -v.

JWT Structure

# Header
{ "alg": "HS256", "typ": "JWT" }

# Payload (claims)
{
  "sub": "user@example.com",     ← subject = email (lookup key)
  "roles": ["ROLE_MANAGER"],     ← encoded roles
  "iat": 1716400000,             ← issued at
  "exp": 1716486400              ← expires at (iat + expiry-ms)
}

# Signature
HMACSHA256(base64(header) + "." + base64(payload), secret)
The Role Prefix Convention

Spring Security requires role names to be prefixed with ROLE_ when using hasRole() — it adds the prefix automatically. If you use hasAuthority("ROLE_ADMIN") you provide it yourself. Store roles in the DB as ADMIN (no prefix), convert to ROLE_ADMIN in UserDetailsServiceImpl when building GrantedAuthority objects. Mixing these conventions is a common source of silent 403s.

1. Spring Boot defaults spring.jpa.open-in-view to true. What concrete problem does this cause in a REST API, and why is it particularly hard to notice in development?

With open-in-view=true, a database connection (or at least a Hibernate Session) is held open for the entire HTTP request, including during Jackson serialization of the response. This means that when Jackson walks the object graph to produce JSON, any lazily-loaded association will silently trigger an additional SQL query — you get N+1 queries at serialization time, and nothing in your service layer warned you.

It's invisible in development because H2 is in-memory and blazing fast, so 40 extra queries take nanoseconds. In production with PostgreSQL, load, and a connection pool of 10, this manifests as connection pool exhaustion under load. The unit tests pass, the dev app "works," and you only find out in prod.

2. Why do we store roles in the JWT claims rather than looking them up from the database on every request?

Stateless JWTs are the point — if you hit the database on every request to fetch roles, you've introduced the same latency and single-point-of-failure as a session store without any of its benefits (like immediate revocation).

The tradeoff: role changes don't take effect until the JWT expires. If you promote a user to ADMIN, they need to log in again to get a new token with the updated role. This is acceptable for most apps (use a short expiry) but is the correct trade-off to understand. The alternative — a token blacklist — turns your stateless JWT into a quasi-session system.

3. The project_members join table uses a composite PK of (project_id, user_id). What would force you to promote this into a full ProjectMembership entity with its own surrogate key?

Any time you need to store data about the relationship itself — not about the project, not about the user, but about their connection. Examples: when the user joined, their role within the project (different from their global MEMBER/MANAGER role), invitation status (INVITED, ACCEPTED, DECLINED), or whether they've been deactivated from that specific project.

In JPA terms: you can't put extra columns on a @ManyToMany join table without making it a real entity. When that moment comes, you refactor to a @OneToMany from both sides pointing to the ProjectMembership entity.

4. We chose Flyway over Liquibase. In what scenario would Liquibase be the clearly better choice?

Liquibase wins when you need rollback support or multi-database targeting. Flyway migrations are SQL-first and therefore database-specific — a CREATE TABLE with PostgreSQL-specific types won't run on MySQL. Liquibase's XML/YAML format is database-agnostic: it describes what to create and generates the appropriate SQL per dialect.

The other Liquibase advantage is diff-based migration generation — you can point Liquibase at two database states and it generates the migration to get from one to the other. Flyway requires you to write the migration by hand. For teams doing continuous schema iteration, that automation is valuable.

5. We said "never use @Data on a JPA entity." Explain specifically how @Data's generated equals and hashCode can cause data loss when used with a Set-based collection.

@Data generates equals/hashCode based on all fields. A JPA entity's id field is null before it's persisted (before em.persist() or repository.save() completes). Two new (transient) entities with all other fields equal will be considered equal — and only one will end up in a Set.

Worse: if you add a transient entity to a Set, then persist it (now it has an id), its hash code changes. The Set can no longer find it — lookups and contains() break silently. The correct fix: base equals/hashCode on a stable, non-null business key (like the UUID, once assigned at construction time via UUID.randomUUID() in a field initializer) or use Hibernate's recommended @NaturalId approach.

6. Why is CSRF protection explicitly disabled in this API's SecurityConfig, and under what circumstances would you need to re-enable it?

CSRF attacks exploit the browser's automatic cookie-sending behavior: a malicious site tricks the browser into making a request to your API using the user's session cookie. A stateless JWT API doesn't use cookies for authentication — the Authorization: Bearer ... header is never sent automatically by the browser. Therefore there's nothing to hijack, and CSRF protection provides no benefit.

You'd re-enable it if you switched to cookie-based authentication (e.g., storing the JWT itself in an HttpOnly cookie for security against XSS, or using Spring Session). The moment the browser automatically sends your auth credential, CSRF becomes a real threat again. This is why the "JWT in HttpOnly cookie" pattern requires CSRF tokens — it combines the XSS resilience of HttpOnly cookies with CSRF protection via double-submit or synchronizer tokens.

How Spring Boot Auto-Configuration Actually Works

This is the most important concept to internalize before writing any beans yourself. When Spring Boot sees a dependency on the classpath, it activates matching auto-configuration classes — each one is a conditional @Configuration that only fires when certain conditions hold.

The Chain for H2 + JPA in the Test Run

  spring-boot-starter-data-jpa on classpath
       │
       ▼
  HibernateJpaAutoConfiguration fires
  (@ConditionalOnClass Hibernate + JPA present)
       │
       ├─ DataSourceAutoConfiguration fires first
       │    @ConditionalOnMissingBean(DataSource)  ← no explicit DS configured
       │    @ConditionalOnEmbeddedDatabase         ← H2 is on the classpath
       │    → creates embedded H2 DataSource automatically
       │
       ├─ HikariCP is the default pool
       │    HikariDataSource wraps the H2 connection
       │
       └─ Hibernate initializes against that DataSource
            spring.jpa.hibernate.ddl-auto defaults to "create-drop"
            for embedded databases (when not explicitly set)
            → schema created, then dropped after tests

This is why the context test passed without any explicit datasource configuration — the auto-configuration chain figured out what to do from the classpath alone. The relevant source to browse when curious: spring-boot-autoconfigure JAR → META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. Every auto-configuration class is listed there.

The Condition Hierarchy

@ConditionalOnMissingBean is the key pattern. Every auto-configuration bean is wrapped in this condition, meaning: if you define your own bean of this type, mine steps aside. This is how you customize: define your own DataSource bean and Spring Boot's auto-configuration defers to it without any special opt-out code.

@SpringBootApplication — What's Actually Inside

@SpringBootConfiguration   // marks this class as the source of bean definitions
@EnableAutoConfiguration    // activates the auto-config chain described above
@ComponentScan              // scans com.example.taskflow and all sub-packages

@SpringBootConfiguration is essentially @Configuration — it tells Spring this class can declare @Bean methods. @EnableAutoConfiguration reads the imports file and conditionally activates all the auto-configs. @ComponentScan without arguments scans from the package of the annotated class — which is why the main class lives at the root of your package hierarchy, not inside a sub-package.

@ConfigurationProperties — Type-Safe Config Binding

@Value("${app.jwt.secret}") works but has two problems: it's a string injection (no type coercion, no validation at startup), and it scatters config across every class that uses it. @ConfigurationProperties groups related properties into a single typed object and validates them once at startup. 

@ConfigurationProperties(prefix = "app.jwt")
public record JwtProperties(String secret, long expiryMs) {}

The prefix = "app.jwt" maps to app.jwt.secret and app.jwt.expiry-ms (Spring's relaxed binding converts expiry-ms → expiryMs automatically — kebab-case in config, camelCase in Java).

@ConfigurationPropertiesScan on the main class tells Spring Boot to scan for all @ConfigurationProperties classes and register them as beans. Without this (or @EnableConfigurationProperties(JwtProperties.class)), the record would compile fine but never be bound — a silent miss that only shows up when you try to inject it.

Records + ConfigurationProperties Gotcha

Java records are immutable by design. Spring Boot binds @ConfigurationProperties to records via the canonical constructor — every component in the record header maps to a config property. This works in Spring Boot 2.6+ with records declared exactly like the example above. If you add a custom constructor, Boot may get confused about which one to use; stick to the canonical form.

Adding Validation Later (Module 5)

When we wire up JWT in Module 5, we'll add @Validated to the record and @NotBlank / @Min on the components. Spring Boot will then validate the bindings at startup and fail fast with a clear error message rather than an NPE at the first request.

Spring Security on the Classpath — What Happens Automatically

The moment spring-boot-starter-security appears in the pom, Spring Boot's SecurityAutoConfiguration fires and locks down every endpoint with HTTP Basic authentication. A random password is printed to the console at startup. This surprised the entire Java world when Boot first introduced it — it's a "secure by default" decision that is correct for production but painful for development.

Our SecurityConfig overrides this default entirely: 

@Bean

Why @EnableMethodSecurity Now?

This annotation activates Spring Security's AOP-based method-level security (@PreAuthorize, @PostAuthorize, @Secured). Putting it here in Module 1 means all future @PreAuthorize annotations we write in controllers and services will be processed without any additional wiring. Forgetting it and adding it later is a common gotcha — the annotations silently do nothing without it.

Why PasswordEncoder and AuthenticationManager Now?

Both are infrastructure beans that will be injected by Module 5's AuthService and UserDetailsServiceImpl. Defining them here, in SecurityConfig, avoids circular dependency issues later. Spring Security's AuthenticationManager is tricky to expose as a bean — config.getAuthenticationManager() is the idiomatic way to do it in Spring Security 6+/7.

HikariCP — The Connection Pool You're Getting for Free

Spring Boot defaults to HikariCP as the connection pool since Boot 2.0. It's measurably the fastest JDBC pool in the Java ecosystem. The key settings in application-prod.properties:

PropertyValueWhy
maximum-pool-size20Max concurrent DB connections. Rule of thumb: (2 × CPU cores) + effective spindle count. For a typical 4-core cloud VM, 10–20 is right. More is worse, not better.
minimum-idle5Connections kept open even when idle. Reduces latency on cold traffic.
connection-timeout30000msHow long to wait for a connection from the pool before throwing. 30s is the default; keep it.

Pool Exhaustion — the Invisible Killer

If maximum-pool-size is set too high and your DB can't handle that many concurrent connections, the DB falls over, not the app. If it's set too low and a slow query holds a connection while 20 other threads are waiting, you get a wall of HikariPool-1 - Connection is not available, request timed out after 30000ms errors. The correct approach: measure actual concurrent usage under load, then set the pool size to match. Never set it to a high number because "more is safer" — it isn't.

2. What is relaxed binding in Spring Boot property resolution, and what does it mean for a property named app.jwt.expiry-ms in the config file vs expiryMs in the record?

Relaxed binding means Spring normalizes property names before matching them to Java fields. The following all bind to the same expiryMs field:

  • app.jwt.expiry-ms (kebab-case — preferred in .properties files)
  • app.jwt.expiryMs (camelCase)
  • app.jwt.expiry_ms (underscore)
  • APP_JWT_EXPIRY_MS (environment variable form — uppercase + underscores)

This is particularly important for Docker/Kubernetes deployments where environment variables use uppercase and underscores but your config file uses kebab-case. DB_URL in the environment correctly overrides spring.datasource.url... but that's different from relaxed binding — that's Spring's env var property source. The relaxed binding applies within a single property source (e.g., within a .properties file).

3. SessionCreationPolicy.STATELESS is set in the security config. What does this prevent Spring Security from doing, and what breaks if you forget it while using JWT?

Without STATELESS, Spring Security's default is IF_REQUIRED — it creates an HttpSession and stores the Authentication object in it. This means:

  • The first valid JWT request creates a session. Subsequent requests from the same client are served from the session, not from the JWT.
  • If you revoke a JWT (or issue a new one after a password change), the old session keeps the user authenticated.
  • Your API is no longer stateless — it can't be horizontally scaled without sticky sessions or a shared session store.

STATELESS tells Spring Security: never create or use an HttpSession. Every request must carry its own credentials. This is the correct mode for Bearer-token APIs.

4. The PasswordEncoder and AuthenticationManager beans are defined in SecurityConfig rather than in a future AuthService. Why does defining them later in AuthService cause a circular dependency in Spring?

The circular dependency chain would be:

AuthService needs AuthenticationManager → AuthenticationManager is built by Spring Security using UserDetailsService → UserDetailsServiceImpl needs UserRepository (fine) but also needs PasswordEncoder → and you defined PasswordEncoder in AuthService… which hasn't finished initializing yet.

In practice the exact cycle depends on your implementation, but the root issue is that Spring Security's AuthenticationManagerBuilder is itself a Spring bean that gets autowired during security configuration — trying to inject it into an application-level bean that is also part of that configuration creates a circular graph Spring can't resolve.

The fix is to define PasswordEncoder and AuthenticationManager in a pure infrastructure config class (SecurityConfig) that has no application-layer dependencies. Everything else depends on it; it depends on nothing.

5. The docker-compose.yml uses a named volume (pgdata) rather than a bind mount (./data:/var/lib/postgresql/data). What are three concrete reasons to prefer the named volume for local development?
  • No accidental commits: A bind-mounted ./data/ directory will show up in git status. Even with a good .gitignore entry, new team members may accidentally stage it. Named volumes live entirely in Docker's managed storage and are invisible to git.
  • No IDE indexing: IntelliJ and VS Code aggressively index project directories. A ./data/ full of PostgreSQL binary files can cause IDE slowdowns and spurious search results. Named volumes avoid this entirely.
  • No permission issues on macOS/Linux: PostgreSQL runs as the postgres user inside the container (UID 999 typically). Bind mounts on macOS involve Docker Desktop's VM layer and can produce ownership mismatches that cause Postgres to refuse to start. Named volumes are owned and managed by Docker and don't have this problem.

JPA Entity Design Decisions

UUID Primary Keys — and Why They're Set by JPA, Not the DB

@GeneratedValue(strategy = GenerationType.UUID) is the JPA 3.1 standard (Hibernate 6+). Hibernate generates the UUID in-memory, before the INSERT — this is different from a database sequence or SERIAL column, where the ID only exists after the row is written. In-memory generation means:

  • The entity has a non-null id immediately after em.persist() is called — no extra SELECT needed
  • You can reference the entity by ID before flushing
  • The DB column has no DEFAULT gen_random_uuid() needed — that default in the Flyway script is only for direct SQL inserts that bypass JPA

LAZY Fetching on Every Relationship

Every @ManyToOne in the codebase has fetch = FetchType.LAZY. This is a production-critical decision. The JPA default for @ManyToOne is actually EAGER, which means: 

  taskRepository.findById(taskId)
       │
       └─ EAGER (bad default)
           ├─ SELECT * FROM tasks WHERE id = ?
           ├─ SELECT * FROM projects WHERE id = ?    ← always loaded, even if unused
           ├─ SELECT * FROM users WHERE id = ?       ← reporter, always loaded
           └─ SELECT * FROM users WHERE id = ?       ← assignee, always loaded

  taskRepository.findById(taskId)
       │
       └─ LAZY (explicit, correct)
           └─ SELECT * FROM tasks WHERE id = ?
              (project, reporter, assignee are proxies — only loaded on access)

With open-in-view=false and LAZY everywhere, you'll get a LazyInitializationException the moment you access an association outside a transaction. That is the correct behavior — it forces you to either load what you need up front with a JOIN FETCH query, or structure your code so the access happens inside a @Transactional boundary.

@Enumerated(EnumType.STRING) — Never ORDINAL

EnumType.ORDINAL stores the zero-based index of the enum constant (TODO = 0, IN_PROGRESS = 1, etc.). The moment you insert a new value between existing ones or reorder the enum, every stored value becomes wrong — silently, with no error. Use STRING always. The storage cost difference (a few bytes per row) is irrelevant.

Hibernate 7 Maps Enums to the SQL ENUM Type

When running against H2 in the test suite, the actual DDL Hibernate 7 generated was: 

status enum ('ACTIVE','ARCHIVED') not null
priority enum ('CRITICAL','HIGH','LOW','MEDIUM') not null

This is new behavior in Hibernate 7 — it uses the database's native ENUM type when @Enumerated(EnumType.STRING) is detected. In H2 this is a genuine ENUM-like constraint. For PostgreSQL, Hibernate 7 generates a CHECK constraint on the VARCHAR column rather than creating a PostgreSQL ENUM type (which would require DDL to alter when the enum grows). Our Flyway migrations use VARCHAR(50) which Hibernate's ddl-auto=validate accepts.

The equals/hashCode Contract for JPA Entities

Lombok's @Data generates equals/hashCode from all fields. On a JPA entity, this is a correctness bug. The recommended pattern: 

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User other)) return false;
    return id != null && id.equals(other.id); // null-safe: two new entities never equal each other
}

@Override
public int hashCode() {
    return getClass().hashCode(); // constant — survives transient→managed transition
}

The constant hashCode is the key insight. If you return id.hashCode() (null before persist), the entity's hash changes when it gets its ID — any Set that held it can no longer find it. A constant hashCode means every instance of the same class maps to the same bucket, which is inefficient for large sets but correct for all lifecycle states. For small domain sets (members of a project) this is perfectly acceptable.

The @ManyToMany List vs Set Problem ("Bag Delete")

The Project.members collection uses Set<User>, not List<User>. If you use a List with @ManyToMany, Hibernate treats the collection as a "bag" (unordered, allows duplicates). When you remove a single member from a bag, Hibernate's strategy is:

  1. DELETE all rows from project_members for this project
  2. Re-INSERT every remaining member

A project with 100 members, removing one, generates 101 SQL statements. A Set removes exactly 1 row. Use @Builder.Default alongside @Builder to ensure the field initializer (= new HashSet<>()) is respected by the generated builder — without it, @Builder sets the field to null.

Spring Data JPA — How Derived Queries Work

Spring Data parses the method name at startup and generates JPQL (and then SQL) from it. The parsing is keyword-based:

Method nameGenerated JPQL
findByEmail(String email)SELECT u FROM User u WHERE u.email = :email
existsByEmail(String email)SELECT count(u.id) > 0 FROM User u WHERE u.email = :email
findByMembersContaining(User user)SELECT p FROM Project p JOIN p.members m WHERE m = :user
findByTaskOrderByCreatedAtAsc(Task task)SELECT c FROM Comment c WHERE c.task = :task ORDER BY c.createdAt ASC
countByProjectAndStatus(Project p, TaskStatus s)SELECT count(t) FROM Task t WHERE t.project = :p AND t.status = :s

The method name grammar is: [find|count|exists|delete]By[Property][Operator] with And/Or for conjunctions and OrderBy[Property][Asc|Desc] as a suffix. If the method name grows complex or requires a JOIN FETCH, switch to @Query("SELECT ...") — don't torture the derived query parser beyond 2–3 conditions.

Pagination via Pageable

Any repository method that accepts a Pageable parameter automatically generates a LIMIT / OFFSET query and returns a Page<T> with total count metadata. The controller passes in PageRequest.of(page, size, Sort.by("createdAt").descending()). The Page wrapper gives you getTotalElements(), getTotalPages(), hasNext() — everything a pagination UI needs.

The Count Query Gotcha with JOIN FETCH and Pagination

When Spring Data paginates a query that has a JOIN FETCH (e.g., to eagerly load an association), it has to run a separate COUNT query to compute totalElements. If the main query has a JOIN FETCH, Hibernate cannot derive the count query automatically and throws a runtime exception. The fix: provide an explicit countQuery attribute in the @Query annotation. This is a common "works in dev, breaks in prod" issue because it only manifests when you actually try to paginate.

Flyway Migrations — Structure and Conventions

Naming Convention

The double underscore separates version from description. Single underscore is a typo that Flyway silently ignores — the file just won't execute. Always use double.

What Flyway Does at Startup

  Application starts (prod profile)
       │
       ▼
  Flyway scans classpath:db/migration
       │
       ▼
  Reads flyway_schema_history table  ← created automatically on first run
       │
       ├─ V1 already applied? Skip
       ├─ V2 already applied? Skip
       ├─ V3 not applied? Run it, record in flyway_schema_history with checksum
       └─ V4 not applied? Run it, record with checksum
       │
       ▼
  Hibernate ddl-auto=validate
  ← confirms entity model matches the schema Flyway just created

The checksum is the critical safety mechanism: Flyway hashes the content of every applied migration. If you modify a migration after it's been run (even a comment change), Flyway throws an error on next startup: "Checksum mismatch for migration V1... Expected: -1234, got: 5678". This prevents the classic mistake of editing a deployed migration to "fix" it. Never modify an applied migration. Add a new one instead.

TIMESTAMPTZ vs TIMESTAMP

PostgreSQL's TIMESTAMP stores the literal date/time with no timezone information. TIMESTAMPTZ (timestamp with time zone) stores the value in UTC and converts on display based on the session's timezone. Always use TIMESTAMPTZ for audit fields — if you ever deploy across timezones or daylight saving boundaries, TIMESTAMP data becomes ambiguous. Hibernate 7 maps Java's Instant to TIMESTAMP(6) WITH TIME ZONE which maps directly to PostgreSQL's TIMESTAMPTZ.

Why em.clear() Matters

@Test
void savesAndRetrievesUserByEmail() {
    em.persistAndFlush(user);
    em.clear();  // ← without this, findByEmail returns the cached in-memory object

    var found = userRepository.findByEmail("alice@example.com");
}

Without em.clear(), the JPA first-level cache (the persistence context) would return the exact same object you just persisted — from memory, without touching the database. Your assertion would pass even if the column mapping was completely wrong and nothing was actually written. em.clear() detaches all entities, forcing the next repository call to execute a real SELECT. This is the difference between testing your code and testing Hibernate's cache.

Questions — reveal answers individually
1. A colleague opens a ticket: "After we added a new value to the TaskStatus enum, all existing CANCELLED tasks now show as IN_REVIEW." What went wrong, and what setting in the codebase prevents this?

The colleague was using @Enumerated(EnumType.ORDINAL). Ordinal storage maps each enum constant to its zero-based position in the Java declaration. If a new value was inserted between IN_REVIEW and DONE, every stored ordinal from that point onward shifted by one — rows that were CANCELLED (was ordinal 4) are now read back as the new value at position 4, which might be IN_REVIEW if the insertion moved things around.

The fix is already in this codebase: @Enumerated(EnumType.STRING) on every enum field. With STRING, Hibernate stores the name ("CANCELLED") and reads it back by name — order doesn't matter, new values can be inserted anywhere in the declaration without affecting stored data.

2. The Project.members field is annotated with both @Builder.Default and initialized to new HashSet<>(). What happens if you remove @Builder.Default while keeping the initializer?

Lombok's @Builder generates a builder that initializes every field to its default Java value (i.e., null for reference types) — it does not use the field initializer expression in the builder path. Without @Builder.Default, any entity created via the builder will have members = null. When Hibernate tries to flush the entity, accessing the null collection (e.g., when cascading or running orphan removal checks) throws a NullPointerException or a Hibernate-level error.

The field initializer (= new HashSet<>()) only runs when the object is created via the no-args constructor (used by Hibernate for loading from DB). Builder-created instances bypass field initializers entirely — hence the need for @Builder.Default.

3. Explain why findByMembersContaining(User user) works as a derived query on a @ManyToMany relationship. What SQL does it produce, and what are the performance implications?

Spring Data detects members as the @ManyToMany collection field in Project. The Containing keyword on a collection generates a JOIN against the join table. The produced SQL is approximately:

SELECT p.*
FROM projects p
INNER JOIN project_members pm ON pm.project_id = p.id
WHERE pm.user_id = ?

Performance implications: this is a simple indexed join (assuming project_members has an index on user_id — our Flyway migration doesn't add one explicitly, though the composite PK (project_id, user_id) only covers lookups by project_id first). For "all projects for user X" queries, you'd want a separate index on project_members(user_id). Worth adding in a V5 migration once this becomes a hot path.

4. The entities use Hibernate's @CreationTimestamp and @UpdateTimestamp. What is the alternative Spring Data approach, and when would you prefer it?

Spring Data's approach uses @EntityListeners(AuditingEntityListener.class) on the entity, combined with @CreatedDate / @LastModifiedDate (and optionally @CreatedBy / @LastModifiedBy for user tracking), and @EnableJpaAuditing on a configuration class.

Prefer the Spring Data approach when you need:

  • User tracking: @CreatedBy / @LastModifiedBy populate from an AuditorAware bean that reads from SecurityContextHolder — giving you "who created/modified this" for free
  • Framework portability: @CreatedDate is a Spring Data annotation, not Hibernate-specific — it works identically with MongoDB, Cassandra, etc. if you ever change the persistence layer

Prefer Hibernate's @CreationTimestamp / @UpdateTimestamp when you only need timestamps (no user tracking) and want zero extra configuration — no @EnableJpaAuditing or listener class required.

5. The @DataJpaTest slice does not load application-dev.properties or the Flyway migration scripts. The schema is created entirely by Hibernate from the entity annotations. @Transactional — The Most Misunderstood Spring Annotation

The Class-Level + Method-Level Override Pattern

@Service
@Transactional(readOnly = true)   // default for every method in the class
@RequiredArgsConstructor
public class ProjectService {

    @Transactional                    // overrides to readOnly=false for writes
    public Project create(...) { ... }

    public Project findById(UUID id) { ... }  // inherits readOnly=true
}

readOnly = true is not just a hint — it does real work:

  • Hibernate skips dirty checking at flush time. In a read-only transaction, Hibernate doesn't take a snapshot of loaded entities and doesn't compare their current state to that snapshot before the transaction closes. For queries that load dozens of objects, this is a measurable CPU saving.
  • Some JDBC drivers and databases route to a read replica. Spring's LazyConnectionDataSourceProxy + a read/write split datasource will route readOnly=true transactions to a replica automatically.
  • It documents intent. A future engineer reading readOnly=true knows immediately: this method must not have side effects. The annotation does what the name says.

How @Transactional Actually Works (AOP Proxy)

  Controller calls projectService.create(...)
        │
        ▼
  Spring AOP proxy intercepts the call
        │
        ├─ opens a transaction (or joins existing)
        ├─ acquires a DB connection from HikariCP pool
        │
        ▼
  Real ProjectService.create() runs
        │

The Self-Invocation Trap

Because @Transactional works via a proxy, calling a transactional method from within the same class bypasses the proxy entirely — the annotation is silently ignored. This is the most common @Transactional bug in production: 

public class ProjectService {
    public void doSomething() {
        this.create(...); // ← @Transactional on create() is IGNORED — direct call, not through proxy
    }
}

The fix: inject ProjectService into itself (ugly), use ApplicationContext.getBean() (worse), or restructure so the transactional boundary is never crossed internally.

Why RuntimeException — Not Checked Exception — Triggers Rollback

Spring's default rollback rule: roll back on any unchecked exception (RuntimeException or Error), commit on checked exceptions. This is a deliberate design choice from the early 2000s: checked exceptions were considered "expected, recoverable" situations (e.g., IOException from a file that might not exist) while unchecked exceptions represented programming errors or unrecoverable failures.

All four of our exceptions extend RuntimeException, which means any service method that throws them will trigger an automatic rollback — no partial writes left in the DB. If you used checked exceptions, you'd need to explicitly annotate: @Transactional(rollbackFor = SomeCheckedException.class). The Spring convention is to use unchecked exceptions for application-level errors and let rollback happen automatically.

The Exception Hierarchy — HTTP Status Mapping

Each exception class carries implicit semantic meaning that the global error handler (Module 4's GlobalExceptionHandler) will translate to an HTTP status code. The mapping is defined once, centrally:

ForbiddenException403 ForbiddenCaller lacks permission for this specific resource

The distinction between 403 and 422 matters: a 403 means "you can't do this" (identity/permission issue); a 422 means "even with permission, this operation doesn't make sense" (domain rule violation). A task status transition from DONE → TODO is always invalid regardless of who's asking — that's a 422, not a 403.

State Machine in the Enum

The status transition rules live in TaskStatus, not in TaskService. This is a deliberate choice about where domain logic belongs. 

public boolean canTransitionTo(TaskStatus next) {
    return switch (this) {
        case TODO        -> Set.of(IN_PROGRESS, CANCELLED).contains(next);
        case IN_PROGRESS -> Set.of(IN_REVIEW, TODO, CANCELLED).contains(next);
        case IN_REVIEW   -> Set.of(DONE, IN_PROGRESS, CANCELLED).contains(next);
        case DONE, CANCELLED -> false;
    };
}

Why a Switch Expression, Not Abstract Methods

The alternative — abstract method per constant with anonymous class bodies — is more object-oriented but has a critical flaw: Java enum constants with anonymous class bodies use a different class per constant, which breaks serialization, reflection-based tooling, and some framework integrations (Jackson, Hibernate). A switch expression keeps the enum as a single concrete class with one flat method. The compiler enforces exhaustiveness on the switch — if you add a new TaskStatus constant and forget to handle it here, the build fails. That's the guarantee you want from a state machine.

The Service Calls It

public Task updateStatus(UUID taskId, TaskStatus newStatus) {
    var task = findById(taskId);

        throw new BusinessRuleException(
            "Cannot transition task from %s to %s".formatted(task.getStatus(), newStatus));
    }
    task.setStatus(newStatus);
    return taskRepository.save(task);
}

The service enforces the rule; the enum defines it. If the rule changes, there's exactly one place to update.

  • Testability: you can instantiate the class in a plain unit test without a Spring context — just pass mocks to the constructor. @InjectMocks in Mockito does exactly this.
  • Immutability: final fields cannot be reassigned after construction, which matters for thread safety in singleton beans.
  • Circular dependency detection: Spring detects circular constructor injection at startup and throws an explicit error. With field injection, it silently creates half-initialized beans at runtime.

Module 3 Quiz

Questions — reveal answers individually
1. A service method annotated with @Transactional(readOnly = true) calls entity.setSomeField("value") and returns. Is the change persisted? Why or why not?

It depends on the Hibernate flush mode. With readOnly = true, Spring sets the Hibernate session to FlushMode.MANUAL, which means Hibernate will not automatically flush dirty entities at transaction close. So in practice, the change will not be persisted — Hibernate skips the dirty check entirely.

However: if you explicitly call entityManager.flush() inside a readOnly transaction, Hibernate will attempt to flush and the DB driver may throw an error (read-only connection). The behavior is technically "Hibernate won't do it automatically," not "it's physically impossible."

The production risk: a developer loads an entity in a readOnly method, mutates it, and the change is lost silently (no exception, no log). This is a footgun — which is why the readOnly = true at class level + explicit @Transactional on write methods makes the intent machine-checkable.

2. ProjectService.addMember calls project.getMembers().add(user) and then projectRepository.save(project). But save() on an already-managed entity in a transaction is effectively a no-op — Hibernate will flush the dirty state anyway. So why call save() at all?

You're right that for a managed entity inside a transaction, Hibernate's dirty checking will pick up the change and generate the INSERT into project_members without an explicit save() call. The explicit save() is defensive and documentary:

  • Explicit intent: A reader sees save() and knows a write is happening. Without it, the mutation is implicit and easy to miss in code review.
  • Works outside a transaction too: If the method were called without a surrounding transaction (which shouldn't happen, but can via tests or misconfiguration), the entity would be detached and Hibernate would throw a DetachedObjectException on flush. Calling save() on a detached entity re-attaches it via merge().
  • Returns the managed instance: save() returns the managed entity. Some patterns (especially with Spring Data's @Version optimistic locking) need the returned instance, not the original.
3. The services use @Transactional but do no authorization checks (beyond a few domain rules). Where will authorization live, and how does it get the currently authenticated user without being passed explicitly?

Authorization lives in two places in Module 5:

  1. Method-level: @PreAuthorize annotations on controller or service methods. e.g., @PreAuthorize("hasRole('ADMIN') or @projectSec.isOwner(authentication, #projectId)"). Spring Security evaluates this before the method runs — the method body never executes if the check fails.
  2. Programmatic: When the rule is too complex for SpEL, the service can call SecurityContextHolder.getContext().getAuthentication() to get the currently authenticated principal.

The authenticated user is stored in SecurityContextHolder — a thread-local (or coroutine-scoped) holder that the JwtAuthenticationFilter populates on every request. Any code on the same thread can read it without it being passed as a parameter. This is why service methods don't need a User requestingUser parameter in most cases — they pull the principal from the context when needed.

4. TaskService.delete() calls existsById() before deleteById(). What race condition exists in this pattern, and what's the correct approach to eliminate it?

The check-then-act pattern has a TOCTOU (Time-of-Check-to-Time-of-Use) race: between existsById() returning true and deleteById() executing, another thread or process could have deleted the same row. deleteById() would then silently succeed (deleting 0 rows) without notifying the caller that the resource was already gone.

The correct approach: skip the existence check and use the return value of the delete operation itself. Spring Data's deleteById throws EmptyResultDataAccessException (or similar) if the entity doesn't exist — or you can use a custom @Query("DELETE FROM Task t WHERE t.id = :id") and check the updated count:

@Modifying
@Query("DELETE FROM Task t WHERE t.id = :id")
int deleteByIdReturningCount(@Param("id") UUID id);

// In service:
if (taskRepository.deleteByIdReturningCount(taskId) == 0) {
    throw new ResourceNotFoundException("Task", taskId);
}

This is a single atomic operation — no window for a race condition.

5. The TaskStatus switch expression is exhaustive — the compiler enforces that every enum constant has a branch. What happens at runtime if a future developer adds a new status (e.g., BLOCKED) to the enum but forgets to update the switch?

The code will fail to compile — not at runtime, at build time. Java's switch expressions (using -> arrow syntax) require exhaustive coverage of all enum constants when there is no default branch. Adding BLOCKED to TaskStatus without adding a case BLOCKED to the switch produces a compile error: "the switch expression does not cover all possible input values."

This is the key advantage over a classic if/else if chain or a switch statement with a default — the compiler becomes a safety net. The developer must decide what transitions BLOCKED allows before the code can ship. This is the "make illegal states unrepresentable" principle applied to the build toolchain.

Java Records as DTOs

A record is a special class declaration where the compiler auto-generates the constructor, getters (named after the component), equals, hashCode, and toString. Because all state is declared in the header and there are no setters, records are inherently immutable — making them ideal request/response shapes that should never mutate after construction.

// Compact. Immutable. Compiler-enforced completeness.
public record UserResponse(
        UUID id,
        String email,
        String displayName,
        Role role,
        Instant createdAt
) {
    public static UserResponse from(User user) {
        return new UserResponse(user.getId(), user.getEmail(),
                user.getDisplayName(), user.getRole(), user.getCreatedAt());
    }
}

The static from() factory keeps mapping logic inside the DTO itself — no separate mapper class needed for a project this size. The alternative (MapStruct) generates bytecode at compile time and is worth adding when you have 20+ DTOs or complex nested mappings, but it's overkill here.

Why not Lombok @Value? @Value generates a final class with an all-args constructor and no setters — functionally similar. But it's still an annotation processor layer on top of a class. Records are a first-class language feature: reflection, pattern matching (instanceof deconstruction), and sealed interface implementations all work cleanly with records but need workarounds with @Value.
Why not use entities directly as responses? Three reasons. (1) Lazy associations — Jackson serializes getters, which triggers Hibernate to open a connection outside the transaction (we blocked this with open-in-view=false, so it would throw instead). (2) Leaking internals — passwordHash, updatedAt, internal IDs you never want exposed. (3) API stability — if the entity changes shape (column renamed, field moved), the API contract breaks silently without a DTO layer.

Bean Validation with @Valid

The Jakarta Bean Validation spec (jakarta.validation) defines constraint annotations. Spring MVC activates them when you put @Valid on a @RequestBody parameter. If any constraint fails, Spring throws MethodArgumentNotValidException before your controller method body executes.

// On the DTO record — constraints live where the shape is defined
public record CreateUserRequest(
        @Email @NotBlank String email,
        @NotBlank @Size(min = 2, max = 100) String displayName,
        @NotBlank @Size(min = 8) String password,
        Role role   // nullable — defaults to MEMBER in service
) {}

// In the controller — @Valid triggers the check
@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public UserResponse create(@RequestBody @Valid CreateUserRequest req) {
    return UserResponse.from(userService.create(...));
}

Key constraints in this project: @NotNull (rejects null), @NotBlank (rejects null AND whitespace-only strings), @Email (RFC 5321 format check), @Size(min, max) (string/collection length).

@NotNull vs @NotBlank: For a String field, @NotNull passes if the value is "" (empty string). @NotBlank combines not-null + trimmed length > 0. Always use @NotBlank for user-facing string fields.
@Valid vs @Validated: @Valid on a method parameter triggers validation of that object. @Validated on the class enables method-level validation (validating @PathVariable / @RequestParam values directly with constraint annotations). We use only @Valid here since all constraints live on request body records.

ProblemDetail — RFC 7807 Standard Error Responses

Before Spring 6, teams invented their own error body shapes. Every API looked different. RFC 7807 ("Problem Details for HTTP APIs") standardizes it: a JSON object with type (URI identifying the error class), title, status, detail, and optional extensions. Spring 6 ships ProblemDetail as a first-class return type.

// Spring serializes ProblemDetail to RFC 7807 JSON automatically
@ExceptionHandler(ResourceNotFoundException.class)
ProblemDetail handleNotFound(ResourceNotFoundException ex) {
    var pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    pd.setType(URI.create("https://taskflow.example.com/errors/not-found"));
    return pd;
}

// Wire yields JSON:
// { "type": "https://...", "title": "Not Found", "status": 404, "detail": "Task abc not found" }

The type URI is a stable identifier clients can test against. It doesn't need to resolve to a real URL, but it can — pointing to documentation for that error. title is filled automatically from the HTTP status reason phrase.

@RestControllerAdvice — Centralized Exception Handling

@RestControllerAdvice is a combination of @ControllerAdvice (applies to all controllers) and @ResponseBody (return values are serialized to the response body). Methods annotated with @ExceptionHandler intercept exceptions thrown anywhere in the controller layer — including from services called by controllers.

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessRuleException.class)
    ProblemDetail handleBusinessRule(BusinessRuleException ex) {
        return problem(HttpStatus.UNPROCESSABLE_ENTITY, ex.getMessage(), "business-rule");
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        var detail = ex.getBindingResult().getFieldErrors().stream()
                .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
                .collect(Collectors.joining(", "));
        return problem(HttpStatus.BAD_REQUEST, detail, "validation");
    }
}
Exception handler resolution order: Spring picks the most specific handler. If you have handlers for both Exception and ResourceNotFoundException, a ResourceNotFoundException always hits the specific handler. This lets you add a catch-all @ExceptionHandler(Exception.class) for 500 responses without it swallowing known exceptions.

Our exception → HTTP mapping:

Exception HTTP Status Meaning
ResourceNotFoundException404 Not FoundEntity with given ID doesn't exist
BusinessRuleException422 Unprocessable EntityRequest is syntactically valid but violates domain rules (e.g., invalid state transition)
ConflictException409 ConflictDuplicate (e.g., email already registered)
ForbiddenException403 ForbiddenAuthenticated but not authorized
MethodArgumentNotValidException400 Bad RequestBean Validation failed on request body
422 vs 400: Both indicate client error, but they mean different things. 400 is "I can't parse or validate this input at all." 422 is "I understood the input, it's structurally valid, but it violates a business rule." The distinction matters for clients: on 400, fix the request shape. On 422, fix the application state or workflow. Using 422 for state machine violations (like transitioning from DONE) is the right call.

Pageable & PageResponse — Stable Pagination Contract

Spring MVC automatically resolves Pageable parameters from query strings: ?page=0&size=20&sort=createdAt,desc. This is wired by SpringDataWebAutoConfiguration (auto-configured in Spring Boot). You declare the parameter and Spring handles the parsing.

@GetMapping
public PageResponse<TaskResponse> list(
        @PathVariable UUID projectId,
        @RequestParam(required = false) TaskStatus status,
        Pageable pageable) {     // resolved from ?page=0&size=20&sort=...
    var page = status != null
            ? taskService.findByProjectAndStatus(projectId, status, pageable)
            : taskService.findByProject(projectId, pageable);

}

Module 4 — Self-Check

1. You return a Spring Page<TaskResponse> directly from a controller instead of wrapping it in PageResponse. What is the concrete risk?

Spring's Page serializes to JSON with a pageable nested object that contains sort details, offset, and Spring-internal flags. If Spring Data's internal representation changes across versions, your JSON response shape changes — silently breaking any client parsing it. Additionally, the sort field serializes in a Spring-specific format. Wrapping in your own record gives you a stable, documented contract that decouples client code from framework internals.

2. A validation error fires in a request body. Walk through the exact execution path from the HTTP request to the JSON error response.

(1) Spring MVC's RequestMappingHandlerAdapter deserializes the request body via Jackson into the record type. (2) Because @Valid is present, the MethodValidationInterceptor runs the Jakarta Validation engine against the deserialized object. (3) If any constraint fails, Spring throws MethodArgumentNotValidException before your controller method executes. (4) The exception propagates up to GlobalExceptionHandler.handleValidation(). (5) That method builds a ProblemDetail with 400 status and the field error messages. (6) Spring serializes the ProblemDetail as JSON and writes it to the response.

3. What is the difference between 400 Bad Request and 422 Unprocessable Entity, and which one does TaskFlow return for an invalid state machine transition (e.g., DONE → TODO)?

400 means the server cannot parse or validate the request at the syntax/structure level — malformed JSON, missing required field, invalid UUID format. 422 means the request is syntactically and structurally valid, but the semantic content violates a rule. TaskFlow returns 422 for a DONE → TODO transition: the request body is perfectly valid JSON ({"status": "TODO"} parses fine, it passes Bean Validation), but the domain rule enforced by TaskStatus.canTransitionTo() rejects it. The distinction helps clients decide whether to fix the request shape (400) or fix the application state/workflow (422).

4. You add a new unchecked exception RateLimitException to the codebase but forget to add an @ExceptionHandler for it. What does the client receive?

Spring's default exception handling (via DefaultHandlerExceptionResolver and the ResponseEntityExceptionHandler base class) returns a 500 Internal Server Error with a generic ProblemDetail body (status 500, title "Internal Server Error", no detail leaking stack traces). The stack trace is logged server-side but not exposed to the client. This is the safe default — better than returning raw stack traces. The fix is to add a specific @ExceptionHandler(RateLimitException.class) returning 429 Too Many Requests in GlobalExceptionHandler.

5. Why does the project use PATCH /tasks/{id}/status instead of PUT /tasks/{id}/status for updating task status?

PUT semantically means "replace the entire resource at this URL with the provided representation." A PUT to /tasks/{id}/status would imply you're providing a complete replacement for whatever resource lives at that path. PATCH means "apply a partial modification to the resource." Since we're modifying exactly one field (status) on a task that has many other fields (title, description, assignee, dueDate), PATCH correctly signals partial update intent. In practice both would work technically, but PATCH is more semantically honest and helps clients and API tooling understand the contract.

Stateless Authentication with JWT

Traditional session-based auth stores login state on the server (HttpSession). JWT (JSON Web Token) auth stores it on the client: the server issues a signed token at login, and the client presents it on every subsequent request. The server only needs to verify the signature — no DB lookup, no shared session store. This makes it naturally stateless and horizontally scalable.

A JWT has three Base64URL-encoded parts separated by dots: header.payload.signature. The payload (claims) carries the user identity (sub), issue time (iat), and expiry (exp). The signature is an HMAC-SHA256 of header+payload, so any tampering invalidates it.

// JJWT 0.12 API — builder/parser are obtained from Jwts factory methods
// Generate:
Jwts.builder()
    .subject(userDetails.getUsername())   // claim "sub" = email
    .issuedAt(new Date(now))
    .expiration(new Date(now + expiryMs))
    .signWith(signingKey())               // HMAC-SHA256
    .compact();

// Verify + parse:
Jwts.parser()
    .verifyWith(signingKey())
    .build()
    .parseSignedClaims(token)             // throws JwtException on tamper/expiry
    .getPayload();
The signing key must be at least 256 bits for HS256. Store it as a Base64-encoded string in config. Never commit a real secret; use openssl rand -base64 32 to generate one per environment. The dev default in application.properties is a fixed known-bad key — it exists only to make the app start without configuration.

Security Layer Architecture

Spring Security is a filter chain. Every HTTP request passes through a series of Filter implementations before reaching the servlet (Spring MVC). Our JWT layer adds one custom filter and wires the rest of Spring Security's built-in filters:

HTTP Request
  │
  ▼
JwtAuthenticationFilter    ← reads "Authorization: Bearer ..." header
  │  validates signature + expiry
  │  loads UserPrincipal from DB
  │  sets SecurityContextHolder.getContext().setAuthentication(...)
  ▼
UsernamePasswordAuthenticationFilter  (skipped — we're stateless)
  ▼
ExceptionTranslationFilter  ← converts AccessDeniedException → 403,
  │                           AuthenticationException → 401
  ▼
FilterSecurityInterceptor / AuthorizationFilter
  │  checks .anyRequest().authenticated()
  │  checks @PreAuthorize expressions
  ▼
DispatcherServlet → Controller → Service

Key insight: the filter chain runs before the servlet. If the JWT filter sets authentication in SecurityContextHolder, downstream filters see an authenticated request. If it doesn't (missing/invalid token), the authorization filter rejects it before the controller ever runs.

OncePerRequestFilter: JwtAuthenticationFilter extends OncePerRequestFilter — Spring guarantees the filter executes exactly once per HTTP request, even if the request is forwarded or included internally. Always extend this instead of raw Filter for security-related filters.

UserPrincipal — Separating Security from Domain

Spring Security's UserDetails interface requires methods like getPassword(), getAuthorities(), isAccountNonExpired(). You could implement these on the JPA User entity, but that mixes security concerns into the domain model — the entity would need to know about Spring Security.

Instead, a wrapper record keeps the separation clean:

// Wraps the domain User — Spring Security never touches the entity directly
public record UserPrincipal(User user) implements UserDetails {

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return List.of(new SimpleGrantedAuthority("ROLE_" + user.getRole().name()));
    }

    @Override public String getPassword() { return user.getPasswordHash(); }

    @Override public String getUsername() { return user.getEmail(); }
    // isAccountNonExpired/Locked/CredentialsNonExpired/Enabled default to true in Spring Security 7
}

In Spring Security 7, the four boolean methods have default true implementations in the UserDetails interface. You only need to override them to support account locking, expiry, or disabling.

Why UserDetailsServiceImpl uses UserRepository directly (not UserService): UserService now depends on PasswordEncoder, which is a bean produced by SecurityConfig. If UserDetailsServiceImpl depended on UserService, and SecurityConfig depended on UserDetailsService (to build AuthenticationProvider), there would be a circular bean dependency at startup. Bypassing the service layer and going straight to UserRepository (which has no security dependency) breaks the cycle.

SecurityConfig — Wiring the Filter Chain

The key design rule: inject all dependencies through bean method parameters, not through class-level constructor injection. This lets Spring resolve them lazily during context startup, avoiding circular dependency errors.

@Bean
public SecurityFilterChain filterChain(HttpSecurity http,
        JwtAuthenticationFilter jwtAuthFilter,
        AuthenticationProvider authProvider) throws Exception {
    http
        .csrf(AbstractHttpConfigurer::disable)          // stateless = no CSRF needed
        .sessionManagement(s ->
            s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/v1/auth/**").permitAll()
            .anyRequest().authenticated())
        .authenticationProvider(authProvider)
        .addFilterBefore(jwtAuthFilter,
            UsernamePasswordAuthenticationFilter.class);
    return http.build();
}

@Bean
public AuthenticationProvider authenticationProvider(UserDetailsService uds,
                                                           PasswordEncoder pe) {
    // Spring Security 7: DaoAuthenticationProvider requires UserDetailsService in constructor
    var provider = new DaoAuthenticationProvider(uds);
    provider.setPasswordEncoder(pe);
    return provider;
}

DaoAuthenticationProvider is what AuthenticationManager.authenticate() calls internally. It loads the user via UserDetailsService.loadUserByUsername() and then verifies the password against PasswordEncoder.matches(). You never call it directly — you call authManager.authenticate(new UsernamePasswordAuthenticationToken(email, rawPassword)) and let the provider chain handle it.

@PreAuthorize — Method-Level Authorization

@PreAuthorize evaluates a SpEL (Spring Expression Language) expression before the method executes. If the expression is false, Spring throws AccessDeniedException → 403. @EnableMethodSecurity in SecurityConfig activates this.

// Simple role check
@PreAuthorize("hasRole('ADMIN')")
public UserResponse create(@RequestBody @Valid CreateUserRequest req) { ... }

// Delegate to a service bean — @projectSecurity is the bean name
// #id binds to the @PathVariable UUID id method parameter
// authentication.name calls authentication.getName() = UserPrincipal.getUsername() = email
@PreAuthorize("@projectSecurity.isOwner(#id, authentication.name) or hasRole('ADMIN')")
public ProjectResponse archive(@PathVariable UUID id) { ... }

How authentication.name resolves to the email: SecurityContextHolder holds the UsernamePasswordAuthenticationToken set by the filter. AbstractAuthenticationToken.getName() checks if the principal implements UserDetails and returns getUsername() — which is the email in UserPrincipal. The result: authentication.name in any SpEL expression is always the email of the logged-in user.

Requires -parameters compiler flag for #paramName to work. Spring Boot's spring-boot-starter-parent adds this flag to maven-compiler-plugin automatically. Without it, the JVM discards parameter names at compile time and #id/#projectId in SpEL expressions would fail at runtime. This is a silent gotcha when setting up a non-Boot Maven project.
@PreAuthorize vs filter-layer rules: Filter-layer rules (.anyRequest().authenticated()) run before the controller method. @PreAuthorize runs after the filter but before the method body. Use filter rules for coarse-grained access (authenticated vs not). Use @PreAuthorize for fine-grained checks that require business context (is this user a member of this specific project?).

Auth Flow End-to-End

Registration: POST /api/v1/auth/register → AuthService.register() → UserService.create() hashes password with BCrypt → saves user → JwtService.generateToken() → returns AuthResponse(token, user)

Login: POST /api/v1/auth/login → AuthService.login() → authenticationManager.authenticate(email, rawPassword) → DaoAuthenticationProvider calls loadUserByUsername + matches() → on success, load User, generate token → returns AuthResponse

Authenticated request: Client sends Authorization: Bearer <token> → JwtAuthenticationFilter extracts and validates token → loads UserPrincipal from DB → sets SecurityContextHolder → request reaches controller → @AuthenticationPrincipal UserPrincipal parameter receives the principal

@PostMapping
@ResponseStatus(HttpStatus.CREATED)
@PreAuthorize("@projectSecurity.isMember(#projectId, authentication.name)")
public TaskResponse create(
        @PathVariable UUID projectId,
        @RequestBody @Valid CreateTaskRequest req,
        @AuthenticationPrincipal UserPrincipal principal) {   // injected from SecurityContext
    return TaskResponse.from(taskService.create(
            projectId, principal.user().getId(), ...));

Module 5 — Self-Check

1. A client presents a valid JWT, but the user was deleted from the database after the token was issued. What happens when they make an authenticated request?

JwtAuthenticationFilter validates the signature and expiry — both pass since the token itself is valid. Then it calls userDetailsService.loadUserByUsername(email), which queries the database. If the user no longer exists, UsernameNotFoundException is thrown, which Spring Security translates to a 401. So deletion is reflected within one request. The "vulnerability window" is zero if you load from DB on every request (as we do). If you cached the UserDetails object, you'd have a window equal to the cache TTL.

2. What's the difference between CSRF being "not needed" in a stateless JWT API vs a session-based app, and why is it disabled here?

CSRF (Cross-Site Request Forgery) attacks work by tricking a browser into sending a request to your API using the session cookie the browser automatically includes. The browser has no control over cookie inclusion — it sends them on every matching-domain request. JWT tokens are NOT automatically sent by browsers — the client JavaScript must explicitly read the token from storage and set the Authorization header. A malicious cross-origin page cannot set custom headers on cross-origin requests (CORS blocks it). Therefore, a stateless API that authenticates exclusively via the Authorization header is inherently CSRF-proof, and the CSRF token mechanism would be pure overhead. However: if you ever store the JWT in a cookie (HttpOnly cookie pattern for SSR apps), CSRF becomes relevant again and should be re-enabled.

3. Walk through the exact bean initialization order for SecurityConfig, JwtAuthenticationFilter, UserDetailsServiceImpl, and UserService, explaining why there's no circular dependency.

(1) UserRepository — Spring Data JPA, no spring-managed deps. (2) UserDetailsServiceImpl — needs UserRepository ✓. (3) JwtProperties — @ConfigurationProperties, no deps. (4) JwtService — needs JwtProperties ✓. (5) JwtAuthenticationFilter — needs JwtService ✓ and UserDetailsService (satisfied by UserDetailsServiceImpl) ✓. (6) SecurityConfig beans fire: PasswordEncoder (no deps), then authenticationProvider(uds, pe) (both available), then filterChain(http, jwtAuthFilter, authProvider) (all available), then AuthenticationManager. (7) UserService — needs UserRepository ✓ and PasswordEncoder ✓. (8) AuthService — needs UserService ✓, JwtService ✓, AuthenticationManager ✓. The key design choice: UserDetailsServiceImpl uses UserRepository directly, NOT UserService. If it used UserService, step 2 would need PasswordEncoder, which needs step 6 (SecurityConfig), which needs step 5 (JwtAuthenticationFilter), which needs step 2 — circular.

4. A junior dev adds @Transactional to JwtAuthenticationFilter.doFilterInternal(). What goes wrong?

Filters are not Spring beans managed by the servlet container's filter chain — they're wired via DelegatingFilterProxy, but @Transactional works through AOP proxies. Spring's AOP can proxy a bean, but the proxy is only invoked when the bean is called through Spring's bean machinery. When the servlet container calls doFilter() on the filter, it calls it on the raw object (or through a proxy that doesn't apply @Transactional because filters extend GenericFilterBean, which is not a typical Spring bean proxy target). In practice, @Transactional on a filter method is silently ignored — the annotation has no effect. Additionally, opening a transaction for the duration of the entire filter chain (which includes the entire request lifecycle) is an anti-pattern: it pins a DB connection for the whole request, eliminating the benefit of short-lived transactions. The @Transactional(readOnly = true) on UserDetailsServiceImpl.loadUserByUsername() is the correct placement — it scopes the transaction to just the DB lookup.

5. A user is a member of a project but not its owner. They call DELETE /api/v1/projects/{id}/members/{userId} to remove another member. What response do they get, and at which layer is it decided?

They get 403 Forbidden, decided at the @PreAuthorize layer (method security AOP). The annotation is @PreAuthorize("@projectSecurity.isOwner(#id, authentication.name) or hasRole('ADMIN')"). ProjectSecurityService.isOwner() returns false for a non-owner member, hasRole('ADMIN') is also false, so the expression is false. Spring's method security interceptor throws AccessDeniedException before the controller method body executes. ExceptionTranslationFilter in the filter chain catches it and writes a 403 response. The service layer is never reached — there's no BusinessRuleException or domain check involved. This is the two-layer authorization model: coarse-grained (authenticated?) at the filter chain, fine-grained (is owner?) at the method level.

Tier 1 — Mockito Unit Tests (no Spring)

// @ExtendWith gives you @Mock and @InjectMocks without Spring overhead
@ExtendWith(MockitoExtension.class)
class AuthServiceTest {

    @Mock UserService     userService;
    @Mock UserRepository  userRepository;
    @Mock JwtService      jwtService;
    @Mock AuthenticationManager authenticationManager;
    @InjectMocks AuthService authService;

    @Test
    void login_delegatesToAuthenticationManager_thenReturnsToken() {
        // arrange
        when(userRepository.findByEmail("alice@example.com")).thenReturn(Optional.of(alice()));
        when(jwtService.generateToken(any(UserPrincipal.class))).thenReturn("signed.jwt");

        // act
        var result = authService.login("alice@example.com", "pass1234");

        // assert: verify the delegation happened AND the result is correct
        verify(authenticationManager).authenticate(any(UsernamePasswordAuthenticationToken.class));
        assertThat(result.token()).isEqualTo("signed.jwt");
    }

    @Test
    void login_propagatesBadCredentialsException_beforeLoadingUser() {
        doThrow(new BadCredentialsException("bad")).when(authenticationManager).authenticate(any());

        assertThatThrownBy(() -> authService.login("alice@example.com", "wrong"))
                .isInstanceOf(BadCredentialsException.class);
        // Guard: repo must never be queried if auth failed (saves a DB round-trip)
        verify(userRepository, never()).findByEmail(any());
    }
}

Key patterns

  • verify(mock).method() — asserts a call happened, not just the return value. Use it when the side-effect IS the behavior under test (e.g., a delegation).
  • verify(mock, never()).method() — asserts a call did NOT happen. Useful for confirming fail-fast paths don't do expensive work.
  • doThrow(...).when(mock).method() — the when(...).thenThrow() form doesn't work for void methods; use doThrow instead.
  • any(SpecificType.class) — more precise than bare any(); Mockito's strict stubbing will flag unused stubs as failures.

Why em.clear() matters

Without clear(), Hibernate's first-level (session) cache returns the entity you just persisted — it never hits the database. That means the query you wrote could be completely wrong and your test would still pass. clear() evicts everything, forcing every subsequent load to come from the actual DB. Every @DataJpaTest that tests a find/exists method should call it.

@DataJpaTest vs Flyway

By default, @DataJpaTest replaces the configured DataSource with an embedded H2 and excludes Flyway auto-configuration. Hibernate creates the schema directly from your entity model using ddl-auto=create-drop. This is fast and keeps slice tests isolated, but it means your actual migration SQL is never exercised here — that's the job of the integration tests.

Security Slice Testing: Two Pitfalls and Their Fixes

Pitfall 1 — Wrong 403 for unauthenticated requests

By default, Spring Security uses Http403ForbiddenEntryPoint when no explicit authentication mechanism (form login, HTTP Basic) is configured. REST clients expect 401 Unauthorized (re-authenticate) not 403 (you don't have permission). The fix is adding an explicit HttpStatusEntryPoint to the security config — and this is also a production correctness fix:

// SecurityConfig.java — inside filterChain():
.exceptionHandling(e -> e.authenticationEntryPoint(
        new HttpStatusEntryPoint(HttpStatus.UNAUTHORIZED)))

Pitfall 2 — @WithMockUser ignored in STATELESS sessions

@WithMockUser works by setting the SecurityContextHolder before the test method. But in a stateless JWT setup, Spring Security 7's SecurityContextHolderFilter replaces the holder's content with whatever the RequestAttributeSecurityContextRepository has stored — which is nothing. The mock authentication is silently discarded before any filter runs.

Use SecurityMockMvcRequestPostProcessors.user() instead. It injects the authentication directly as a request attribute, which the RequestAttributeSecurityContextRepository reads during the request — so it survives the filter.

// WRONG — context is cleared by SecurityContextHolderFilter in stateless mode:
@WithMockUser(roles = "ADMIN")
void someTest() { ... }

// CORRECT — injects auth into the request attribute the repository reads:
mvc.perform(get("/api/v1/...")
        .with(user("admin@example.com").roles("ADMIN")))
    .andExpect(status().isOk());

Testing @PreAuthorize with a named bean

@PreAuthorize("@projectSecurity.isMember(...) or hasRole('ADMIN')") calls a bean named projectSecurity via Spring SpEL. In @WebMvcTest, that bean isn't loaded (it's a @Service). Register a mock with the correct bean name:

// name= must match the @Service("projectSecurity") annotation on the real class.
@MockitoBean(name = "projectSecurity") ProjectSecurityService projectSecurity;

Module 6 — Quiz

Q1. You add a new repository method findByStatusAndPriority() and write a @DataJpaTest for it. The test always passes. A colleague points out you forgot a call. What did you forget, and why does it matter?

You forgot em.clear() after em.persistAndFlush(). Without it, Hibernate's first-level (session) cache returns the entity you just saved, never executing the JPQL query you're testing. The query could have a typo or the wrong join, and the test would still pass. clear() evicts everything, forcing a real DB round-trip.

Q2. Your @WebMvcTest test expects 401 from an unauthenticated request, but keeps getting 403. You are importing SecurityConfig which has anyRequest().authenticated(). What's wrong and how do you fix it?

By default, Spring Security uses Http403ForbiddenEntryPoint when there's no explicit authentication mechanism configured (no form login, no HTTP Basic). So unauthenticated requests trigger 403, not 401. Fix: add .exceptionHandling(e -> e.authenticationEntryPoint(new HttpStatusEntryPoint(HttpStatus.UNAUTHORIZED))) to SecurityConfig. This is also a production correctness fix — REST clients use 401 to know they need to re-authenticate.

Q3. A new developer replaces .with(user("alice").roles("ADMIN")) with @WithMockUser(roles="ADMIN") in a @WebMvcTest test. Tests that were passing now return 401 instead of 200. Explain why.

@WithMockUser sets the SecurityContextHolder before the test method runs. But with SessionCreationPolicy.STATELESS, Spring Security 7's SecurityContextHolderFilter replaces the holder's content with whatever is in the RequestAttributeSecurityContextRepository — which is empty for a fresh request. The mock authentication is silently discarded. .with(user(...)) (from SecurityMockMvcRequestPostProcessors) injects auth as a request attribute, which is what the stateless repository reads. Always use the request post-processor form in stateless JWT setups.

Q4. What's the difference between testing @PreAuthorize logic in a @WebMvcTest vs a @SpringBootTest integration test? Which is better for catching auth bugs?

@WebMvcTest is faster (no DB, seconds) and good for verifying the SpEL expression logic itself — does ROLE_ADMIN bypass the member check? Does a non-member get 403? You mock @projectSecurity.isMember() to control the outcome. @SpringBootTest with Testcontainers exercises the real DB: is the member relationship actually stored, does the repository query execute correctly, does the JWT round-trip work? The integration test catches bugs the slice can't. Use both: slices for fast iteration on logic, integration for the full flow.

Q5. Your @DataJpaTest uses H2 and passes. The integration test using Testcontainers PostgreSQL fails with a constraint violation. What class of bug does this reveal?

Schema-code drift or DB dialect differences. H2 is lenient about things PostgreSQL enforces strictly (case sensitivity on identifiers, certain data type conversions, constraint checking). The bug was always there — the slice test couldn't catch it because it ran against Hibernate's auto-generated H2 schema, not your Flyway migration SQL. Only the integration test runs the actual V*.sql files against a real PostgreSQL engine. This is why integration tests exist even when slice tests pass: different execution paths, different failure modes.

Module 7 — Observability

Observability means you can answer "what is the system doing right now, and why?" without attaching a debugger. This module adds three pillars: structured logs, request correlation, and business metrics. All three land in the same pull request because they depend on each other — Micrometer counters are useless if you can't correlate them to a specific request in the logs.

7.1 Structured JSON Logging

Plain text logs like 2026-05-25 INFO UserService - User created are human-readable but machine-hostile. Parsing them with regex is fragile. Structured JSON logs emit one JSON object per log line, where every field is a typed key-value pair that a log aggregator (ELK, Grafana Loki, Datadog) can index without custom parsing.

Spring Boot 4.0 ships built-in structured logging. You pick a format (logstash, ecs, or gelf) and Boot's StructuredLogEncoder handles the serialization. The logstash format is most common and produces fields like @timestamp, level, message, logger_name, and every MDC key as a top-level field.

Profile-aware Logback configuration — logback-spring.xml

The logback-spring.xml file (not logback.xml) is Spring Boot's extension of standard Logback configuration. It gains two powers: <springProfile> blocks that activate based on the active Spring profile, and <springProperty> that reads values from application.properties.

<!-- !prod matches dev, test, and any unnamed profile -->
<springProfile name="!prod">
  <include resource="org/springframework/boot/logging/logback/console-appender.xml"/>
  <root level="INFO">
    <appender-ref ref="CONSOLE"/>
  </root>
</springProfile>

<!-- dev-only DEBUG overrides (stacks on top of !prod block) -->
<springProfile name="dev">
  <logger name="com.example.taskflow"        level="DEBUG"/>
  <logger name="org.springframework.security" level="DEBUG"/>
</springProfile>

<!-- prod: structured JSON to stdout + rolling file -->
<springProfile name="prod">
  <include resource=".../structured-console-appender.xml"/>
  <include resource=".../structured-file-appender.xml"/>
  <root level="INFO">
    <appender-ref ref="CONSOLE"/>
    <appender-ref ref="FILE"/>
  </root>
</springProfile>

Key insight: when the active profile is dev, both the !prod block AND the dev block execute. The dev block's DEBUG overrides the INFO root set by the !prod block for those specific loggers.

Prod format is configured via application-prod.properties:

logging.structured.format.console=logstash
logging.structured.format.file=logstash
logging.file.name=logs/taskflow.log

Spring Boot maps logging.structured.format.console to the Logback property CONSOLE_LOG_STRUCTURED_FORMAT, which the included appender XML reads.

7.2 Request Correlation via MDC

MDC (Mapped Diagnostic Context) is SLF4J's thread-local key-value store. Every log statement emitted during that thread's lifetime automatically includes the MDC contents. This lets you grep all log lines for a single request by filtering on correlationId.

How MDC works

MDC.put("correlationId", "abc-123");   // stored in thread-local map
log.info("Processing request");         // log line includes correlationId=abc-123 automatically
MDC.clear();                            // MUST clear — thread pool reuses threads

Critical: Spring's thread pool (and Tomcat's connection thread pool) reuse threads across requests. If you forget MDC.clear(), the previous request's correlationId leaks into the next request on that thread. Always clear in a finally block.

CorrelationFilter design

@Component
@Order(Ordered.HIGHEST_PRECEDENCE)   // runs before all other filters
public class CorrelationFilter extends OncePerRequestFilter {
    public static final String HEADER = "X-Correlation-ID";

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                    FilterChain chain) throws ServletException, IOException {
        var id = Optional.ofNullable(req.getHeader(HEADER))
                .filter(h -> !h.isBlank())
                .orElseGet(() -> UUID.randomUUID().toString());
        MDC.put("correlationId", id);
        res.setHeader(HEADER, id);           // echo so clients can correlate
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.clear();                     // prevent thread-pool leak
        }
    }
}

@Order(Ordered.HIGHEST_PRECEDENCE) ensures the correlationId is in MDC before Spring Security, your business filters, and your controllers run — so every log statement in the entire request lifecycle carries it.

JwtAuthenticationFilter adds the authenticated user's email to MDC after it resolves the JWT:

SecurityContextHolder.getContext().setAuthentication(authToken);
MDC.put("userId", email);   // cleanup delegated to CorrelationFilter.finally

No separate cleanup needed: CorrelationFilter.finally calls MDC.clear(), which wipes all keys including userId.

Text log pattern for dev/test

# application.properties
logging.pattern.correlation=[%X{correlationId:-?}]

This sets the LOG_CORRELATION_PATTERN Logback variable. Spring Boot's default console appender includes it adjacent to the timestamp. The :-? default means [?] appears when no correlationId is set (e.g., during startup before the first request).

7.3 Business Metrics with Micrometer

Micrometer is Spring Boot's metrics façade — the same idea as SLF4J for logging. You write meterRegistry.counter(...).increment(), and Micrometer forwards it to whatever backend is configured (Prometheus, Datadog, CloudWatch, etc.). Zero code change when you switch backends.

Counters with tags

// TaskService.create()
var saved = taskRepository.save(task);
meterRegistry.counter("tasks.created",
        "priority", saved.getPriority().name().toLowerCase()).increment();
return saved;

// AuthService.register()
meterRegistry.counter("auth.registrations").increment();

Tags are the key feature. tasks.created{priority="high"} and tasks.created{priority="low"} are the same counter name with different dimensions. A Grafana dashboard can show total task creation rate, or break it down by priority, using the same underlying data.

Actuator exposure

# application.properties
management.endpoints.web.exposure.include=health,info,metrics
management.endpoint.health.show-details=when-authorized
management.endpoint.health.show-components=always

GET /actuator/metrics/tasks.created returns the current counter value. GET /actuator/metrics/tasks.created?tag=priority:high filters by tag. In production you'd instead scrape /actuator/prometheus and send to a Prometheus server — but the same counter appears there too.

Mocking Micrometer in service unit tests

Mockito strict mode requires every stub to be consumed. The MeterRegistry.counter varargs overload — counter(String name, String... tags) — needs careful matching. When the production code calls counter("tasks.created", "priority", "medium"), Mockito sees three separate arguments (name + 2 tag strings), not two:

@Mock MeterRegistry meterRegistry;
@Mock Counter counter;

// Correct — match all three expanded vararg elements
when(meterRegistry.counter(anyString(), anyString(), anyString())).thenReturn(counter);

// Wrong — (String[]) any() creates a 2-arg stub, mismatch with 3-arg call
// when(meterRegistry.counter(anyString(), (String[]) any())).thenReturn(counter);

// No-tag counter (AuthService) — simple 1-arg stub
when(meterRegistry.counter("auth.registrations")).thenReturn(counter);

Quiz — Module 7

Q1. Why must MDC.clear() be called in a finally block rather than at the end of the normal request flow?

If an exception propagates out of the filter chain, normal end-of-method code is skipped. The finally block executes regardless of whether chain.doFilter throws. Without it, a thread that handles an error request leaks its MDC into the next request assigned to that thread by the pool — a different request would log with someone else's correlationId.

Q2. What is the difference between <springProfile name="!prod"> and <springProfile name="dev"> in logback-spring.xml?

!prod activates for any profile that is NOT prod — this covers dev, test, and the default (no profile) case. dev activates only when the dev profile is explicitly active. When running in dev, both blocks execute: the !prod block sets up the text console appender, then the dev block overrides specific logger levels to DEBUG. In a test run (no profile), only the !prod block runs — no DEBUG overrides.

Q3. Why are Micrometer counter tags important? What can you do with tasks.created{priority="high"} vs just tasks.created?

Tags add dimensions to a metric without creating separate counters. With a tagged counter, one Grafana query can show you total task creation rate (sum across all tags), and another can break it down by priority to see if high-priority tasks are spiking. Without tags, you'd need separate counter names (tasks.created.high, tasks.created.low) and a separate dashboard panel for each. Tags are composable — you can later add an assignee tag and your existing priority breakdowns still work.

Q4. JwtAuthenticationFilter adds userId to MDC but has no cleanup code. How does the userId get cleared?

CorrelationFilter runs at HIGHEST_PRECEDENCE and wraps the entire filter chain in a try-finally that calls MDC.clear(). Since JwtAuthenticationFilter runs inside that chain, all MDC keys set by any downstream filter — including userId — are cleared when CorrelationFilter.finally executes after the response is complete. Centralising MDC cleanup in one place avoids bugs where individual filters forget to clean up their own keys.

Q5. Why does Mockito require when(meterRegistry.counter(anyString(), anyString(), anyString())) instead of when(meterRegistry.counter(anyString(), (String[]) any())) when stubbing a call like counter("tasks.created", "priority", "medium")?

Java varargs are syntactic sugar: counter("tasks.created", "priority", "medium") compiles to a call with three arguments. Mockito records the arity of the actual invocation (3 args) and compares it to the arity of the stub (2 matchers in the 2-arg form). Since they don't match, Mockito's strict mode raises a PotentialStubbingProblem. Matching each expanded vararg element individually (anyString(), anyString(), anyString()) gives Mockito three matchers for three arguments — arity matches, stub applies correctly.
